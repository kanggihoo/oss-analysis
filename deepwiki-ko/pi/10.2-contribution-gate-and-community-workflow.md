## 기여 Gate 개요

기여 gate는 GitHub Actions로 구현된 보안 및 품질 관리 메커니즘이다. flat-file 권한 시스템과 maintainer 명령을 기반으로 누가 열린 issue와 Pull Request(PR)를 유지할 수 있는지를 제한한다.

### 주요 구성 요소
- **`APPROVED_CONTRIBUTORS`**: GitHub handle과 부여된 capability의 flat-file database [`.github/APPROVED_CONTRIBUTORS:1-5`]().
- **Issue/PR Gates**: 새 issue와 PR을 가로채 gate를 적용하는 자동화 workflow [`.github/workflows/issue-gate.yml:3-5`](), [`.github/workflows/pr-gate.yml:3-5`]().
- **Maintainer Commands**: maintainer가 기여자를 승격하기 위해 사용하는 댓글 keyword `lgtm`과 `lgtmi` [`.github/workflows/approve-contributor.yml:34-37`]().

이 다이어그램은 새 기여자의 첫 제출이 자동화 gate, maintainer 검토, 최종적으로 승인된 기여자 상태로 승격되기까지의 흐름을 보여준다.

## 권한 시스템: APPROVED_CONTRIBUTORS

프로젝트는 기여자 권한을 관리하기 위해 flat text file `.github/APPROVED_CONTRIBUTORS`를 사용한다. 이 파일은 gating workflow의 확정적 source 역할을 한다.

### Capability 수준

| Capability | 설명 |
|------------|-------------|
| `issue` | 해당 사용자의 향후 issue는 열린 상태로 유지된다. PR은 여전히 승인이 필요하다 [`.github/APPROVED_CONTRIBUTORS:4`](). |
| `pr` | 해당 사용자의 향후 issue와 PR이 모두 열린 상태로 유지된다 [`.github/APPROVED_CONTRIBUTORS:5`](). |

파일의 각 줄은 `<username> <capability>` 형식을 따른다 [`.github/APPROVED_CONTRIBUTORS:2`](). `#`로 시작하는 줄은 주석이며 무시된다 [`.github/APPROVED_CONTRIBUTORS:1`]().

### 파싱 로직

workflow들은 `APPROVED_CONTRIBUTORS` 파일을 로드하고 파싱하기 위한 로직을 공유한다. `parseApprovedUsers` 함수는 raw content를 Map으로 처리한다.

```javascript
function parseApprovedUsers(content) {
  const users = new Map();
  for (const rawLine of content.split('\n')) {
    const line = rawLine.trim();
    if (!line || line.startsWith('#')) continue;
    const parts = line.split(/\s+/);
    if (parts.length !== 2) continue;
    const [username, capability] = parts;
    users.set(username.toLowerCase(), capability.toLowerCase());
  }
  return users;
}
```

## 자동화 Gate Workflow

프로젝트는 전용 GitHub Actions workflow를 통해 새 issue와 PR에 gate check를 적용한다.

### Issue Gate(`issue-gate.yml`)

- **트리거:** `issues: opened`에서 실행된다 [`.github/workflows/issue-gate.yml:4-5`]().
- **Bot 제외:** `dependabot[bot]` 같은 bot 계정을 자동으로 건너뛴다 [`.github/workflows/issue-gate.yml:23-26`]().
- **Collaborator 확인:** `github.rest.repos.getCollaboratorPermissionLevel`을 사용해 `admin`, `maintain`, `write` 접근 권한이 있는 사용자를 예외 처리한다 [`.github/workflows/issue-gate.yml:82-86`]().
- **승인 파일 확인:** `getTextFile`과 `parseApprovedUsers`를 통해 작성자를 `APPROVED_CONTRIBUTORS` 목록에 대해 검증한다 [`.github/workflows/issue-gate.yml:88-95`]().
- **적용:** 승인되지 않은 issue에는 댓글을 남기고 닫는다 [`.github/workflows/issue-gate.yml:97-119`]().

### PR Gate(`pr-gate.yml`)

- **트리거:** `pull_request_target: opened`에서 실행된다 [`.github/workflows/pr-gate.yml:4-5`]().
- **엄격한 적용:** issue와 달리 PR은 특별히 `pr` capability가 필요하다. `issue` capability만 있는 사용자의 PR도 자동으로 닫힌다 [`.github/workflows/pr-gate.yml:107-112`]().
- **동작:** PR을 닫고 기여자에게 먼저 issue를 열도록 안내하는 메시지를 게시한다 [`.github/workflows/pr-gate.yml:116-126`]().

### OpenClaw Gate(`openclaw-gate.yml`)

- **목적:** `openclaw/openclaw` repository에서 온 잠재적 자동화 또는 관련 traffic을 식별한다.
- **로직:** 작성자가 `openclaw/openclaw`에서 issue/PR을 연 적이 있는지 확인하기 위해 `github.rest.search.issuesAndPullRequests`를 사용한다 [`.github/workflows/openclaw-gate.yml:93-107`]().
- **Tagging:** 활동이 발견되면 `possibly-openclaw-clanker` label을 추가한다 [`.github/workflows/openclaw-gate.yml:117-122`]().

출처: [`.github/workflows/issue-gate.yml:1-120`](), [`.github/workflows/pr-gate.yml:1-127`](), [`.github/workflows/openclaw-gate.yml:1-123`]()

## Maintainer Workflow: 승격 루프

Maintainer는 특정 명령을 issue에 댓글로 남겨 기여자 권한을 제어한다. 이 명령들은 `approve-contributor.yml` workflow를 트리거한다.

### 승격 명령

| 명령 | 효과 |
|---------|--------|
| `lgtmi` | capability를 `issue`로 설정한다 [`.github/workflows/approve-contributor.yml:34-35`](). |
| `lgtm` | capability를 `pr`로 설정한다 [`.github/workflows/approve-contributor.yml:36-37`](). |

### 승격 로직(`approve-contributor.yml`)

1. **검증:** 댓글 작성자에게 `admin`, `maintain`, `write` 접근 권한이 있는지 확인한다 [`.github/workflows/approve-contributor.yml:51-55`]().
2. **파일 업데이트:** `fs.readFileSync`와 `fs.writeFileSync`를 사용해 `.github/APPROVED_CONTRIBUTORS`를 수정한다 [`.github/workflows/approve-contributor.yml:114-133`]().
3. **Commit:** `github-actions[bot]`이 `chore: approve contributor <username>` 메시지로 변경을 commit한다 [`.github/workflows/approve-contributor.yml:141-144`]().
4. **Push:** default branch에 직접 push한다 [`.github/workflows/approve-contributor.yml:145`]().

## 품질 기준과 지침

프로젝트는 의미 있는 참여를 보장하고 유지보수 부담을 줄이기 위해 엄격한 품질 기준을 유지한다.

### "One Rule"
**자신의 코드를 이해해야 한다.** 구현이나 시스템과의 상호작용을 이해하지 못한 상태로 AI가 생성한 "slop"을 제출하면 즉시 닫힐 수 있다 [`CONTRIBUTING.md:7-9`]().

### Issue 품질
- **Templates:** `bug.yml` 또는 `contribution.yml`을 사용해야 한다 [`CONTRIBUTING.md:30`]().
- **간결성:** issue가 한 화면에 들어가지 않으면 너무 긴 것이다 [`CONTRIBUTING.md:34`]().
- **주말 정책:** 금요일부터 일요일까지 제출된 issue는 자동으로 닫히며, 일반적으로 월요일까지 검토되지 않는다 [`CONTRIBUTING.md:17-19`](), [`CONTRIBUTING.md:79-81`]().

### PR 품질
- **전제 조건:** 이미 `lgtm`으로 승인되지 않았다면 PR을 열지 않는다 [`CONTRIBUTING.md:50`]().
- **검증:** 제출 전에 `npm run check`와 `./test.sh`를 실행해야 한다 [`CONTRIBUTING.md:54-59`]().
- **제한:** `CHANGELOG.md`를 수정하지 않는다 [`CONTRIBUTING.md:61`]().

출처: [`CONTRIBUTING.md:1-94`](), [`.github/ISSUE_TEMPLATE/bug.yml:1-44`](), [`.github/ISSUE_TEMPLATE/contribution.yml:1-37`]()
