이 페이지는 Understand Anything 분석 파이프라인의 Phase 4와 5인 Architecture Analysis와 Tour Building을 자세히 설명합니다. `architecture-analyzer` 에이전트는 intra-group density와 inter-group imports를 기반으로 한 휴리스틱을 사용해 nodes를 논리적 architectural layers에 할당하는 역할을 담당합니다. 이어서 `tour-builder` 에이전트는 Breadth-First Search(BFS)와 fan-in ranking을 활용해 중요한 nodes를 우선순위화하면서 코드베이스를 통과하는 guided learning path를 생성합니다.

## Architecture Analyzer (Phase 4)

`architecture-analyzer` 에이전트는 코드베이스 내의 논리적 architectural layers를 식별하고 모든 파일을 정확히 하나의 layer에 할당합니다 [understand-anything-plugin/agents/architecture-analyzer.md:9-10](). 이 프로세스는 구조 분석을 위한 deterministic script와 semantic interpretation을 위한 LLM 기반 단계, 두 단계로 나뉩니다.

### Phase 1: Structural Analysis Script

첫 번째 단계는 file paths와 import edges를 분석해 structural patterns를 계산하는 스크립트(선호: Node.js, 대체: Python)를 포함합니다 [understand-anything-plugin/agents/architecture-analyzer.md:23-25](). 이 스크립트는 여러 deterministic graph analyses를 수행합니다.

1.  **Directory Grouping**: 공통 path prefix가 제거된 뒤 top-level directory를 기준으로 파일을 그룹화합니다. 구조가 flat하면 file type/extension pattern을 기준으로 그룹화합니다 [understand-anything-plugin/agents/architecture-analyzer.md:54-63]().
2.  **Node Type Grouping**: 코드 파일과 비코드 파일의 분포를 드러내기 위해 file node IDs를 `type`별로 그룹화합니다(예: `file`, `config`, `document`, `service`) [understand-anything-plugin/agents/architecture-analyzer.md:65-68]().
3.  **Import Adjacency Matrix**: 어떤 파일이 다른 파일을 import하는지 보여주는 adjacency list를 구축합니다. 여기에는 각 파일의 fan-out(한 node가 import하는 파일 수)과 fan-in(한 node를 import하는 파일 수), directory groups의 inter-group import relationships 계산이 포함됩니다 [understand-anything-plugin/agents/architecture-analyzer.md:70-74]().
4.  **Cross-Category Dependency Analysis**: 서로 다른 node type groups 사이의 관계를 식별합니다(예: `config` nodes가 `file` nodes를 구성, `service` nodes가 `file` nodes를 배포) [understand-anything-plugin/agents/architecture-analyzer.md:76-87]().
5.  **Inter-Group Import Frequency**: 모든 directory group 쌍 사이의 import edges 수를 보여주는 matrix를 생성하여 dependency direction을 나타냅니다 [understand-anything-plugin/agents/architecture-analyzer.md:89-96]().
6.  **Intra-Group Import Density**: 각 directory group에 대해 해당 group과 관련된 전체 edges 대비 group 내부 import edges의 density를 계산합니다. 높은 intra-group density는 자체 layer에 적합한 cohesive group을 시사합니다 [understand-anything-plugin/agents/architecture-analyzer.md:98-102]().
7.  **Directory Pattern Matching**: directory names를 알려진 architectural patterns와 대조해 분류합니다(예: `routes` -> `api`, `services` -> `service`) [understand-anything-plugin/agents/architecture-analyzer.md:104-150]().

이 스크립트는 계산된 structural patterns를 포함하는 JSON 객체를 출력합니다 [understand-anything-plugin/agents/architecture-analyzer.md:152-169]().

### Phase 2: Semantic Layer Assignment (LLM)

두 번째 단계에서 `architecture-analyzer` LLM 에이전트는 스크립트의 structural insights를 사용해 semantic layer assignments를 수행합니다. 3-10개의 논리적 architectural layers를 식별하고 모든 file node를 정확히 하나의 layer에 할당합니다 [understand-anything-plugin/agents/architecture-analyzer.md:10-11](). language directive가 제공되면 layer names와 descriptions가 번역됩니다 [understand-anything-plugin/agents/architecture-analyzer.md:16-19]().

에이전트는 layer assignment를 위해 다음 요소를 고려합니다.

*   **Directory Grouping**: intra-group density가 높은 cohesive directory groups는 layer의 강력한 후보입니다.
*   **Node Type Grouping**: 유사한 type의 파일(예: 모든 `config` files)은 하나의 layer를 형성할 수 있습니다.
*   **Inter-Group Import Frequency**: 강력한 단방향 의존성을 가진 groups는 종종 계층적 layers를 형성합니다.
*   **Cross-Category Dependency Analysis**: code files를 구성하거나 배포하는 non-code files는 infrastructure 또는 configuration layers로 그룹화될 수 있습니다.
*   **Directory Pattern Matching**: 알려진 architectural patterns(예: `api`, `service`, `data`, `ui`)는 layer naming과 assignment를 안내합니다.

이 단계의 출력은 `Layer` 객체 목록이며, 각 객체는 `id`, `name`, `description`, 그리고 해당 layer에 속한 `nodeIds` 배열을 포함합니다 [understand-anything-plugin/skills/understand-diff/SKILL.md:21]().

## Tour Builder (Phase 5)

`tour-builder` 에이전트는 코드베이스를 통과하는 guided learning tours를 설계하여 프로젝트 아키텍처와 핵심 concepts를 논리적인 순서로 가르치는 5-15개의 교육적 단계를 생성합니다 [understand-anything-plugin/agents/tour-builder.md:10-11](). architecture analyzer와 유사하게 이 단계도 deterministic script와 LLM 기반 component를 포함합니다.

### Phase 1: Graph Topology Script

첫 번째 단계는 tour design에 유용한 structural signals를 드러내기 위해 graph topology를 분석하는 스크립트(선호: Node.js, 대체: Python)를 포함합니다 [understand-anything-plugin/agents/tour-builder.md:25-26]().

스크립트는 다음을 계산합니다.

1.  **Fan-In Ranking (Importance)**: 모든 node에 대해 다른 nodes가 해당 node를 가리키는 edges의 수(fan-in)를 셉니다. 높은 fan-in은 중요성을 나타내며, early understanding에 적합함을 시사합니다 [understand-anything-plugin/agents/tour-builder.md:55-57](). fan-in 기준 상위 20개 nodes가 출력됩니다 [understand-anything-plugin/agents/tour-builder.md:57]().
2.  **Fan-Out Ranking (Scope)**: 모든 node에 대해 해당 node가 다른 nodes를 가리키는 edges의 수(fan-out)를 셉니다. 높은 fan-out은 넓은 범위를 시사하며 overview steps에 유용합니다 [understand-anything-plugin/agents/tour-builder.md:59-61](). fan-out 기준 상위 20개 nodes가 출력됩니다 [understand-anything-plugin/agents/tour-builder.md:61]().
3.  **Entry Point Candidates**: filename patterns(예: `index.ts`, `main.py`), file location(project root 또는 한 단계 아래), 높은 fan-out, 낮은 fan-in을 기반으로 nodes에 점수를 매겨 가능성 높은 entry points를 식별합니다 [understand-anything-plugin/agents/tour-builder.md:63-72](). project root의 `README.md`는 documentation files 중 가장 높은 점수를 받습니다 [understand-anything-plugin/agents/tour-builder.md:74](). 상위 5개 candidates가 출력됩니다 [understand-anything-plugin/agents/tour-builder.md:77]().
4.  **Dependency Chains (BFS from Entry Points)**: 상위 code entry point candidate에서 시작해 `imports`와 `calls` edges를 따라 Breadth-First Search(BFS) traversal을 수행합니다. 이는 코드베이스의 자연스러운 "reading order"를 드러냅니다 [understand-anything-plugin/agents/tour-builder.md:79-83](). BFS traversal order, node depths, depth level별로 그룹화된 nodes가 출력됩니다 [understand-anything-plugin/agents/tour-builder.md:85-87]().
5.  **Non-Code File Inventory**: tour에 포함할 가능성을 위해 non-code files를 category별로 분리합니다(documentation, infrastructure, data/schema, configuration) [understand-anything-plugin/agents/tour-builder.md:89-94]().
6.  **Tightly Coupled Clusters**: 상호 연결성이 높은 2-5개 nodes의 groups를 식별합니다. 이는 함께 설명해야 하는 features 또는 subsystems를 나타냅니다 [understand-anything-plugin/agents/tour-builder.md:96-99]().
7.  **Layer List**: 제공된 layers(id, name, description)를 기록합니다 [understand-anything-plugin/agents/tour-builder.md:106-108]().
8.  **Node Summary Index**: LLM 단계에서 빠르게 참조할 수 있도록 각 node ID에서 해당 `summary`, `type`, `name`으로 이어지는 lookup을 생성합니다 [understand-anything-plugin/agents/tour-builder.md:110-112]().

이 스크립트는 계산된 topological properties를 포함하는 JSON 객체를 출력합니다 [understand-anything-plugin/agents/tour-builder.md:117-130]().

### Phase 2: Pedagogical Tour Design (LLM)

두 번째 단계에서 `tour-builder` LLM 에이전트는 topological insights를 사용해 pedagogical tour를 설계합니다. 각 단계가 `title`, `description`, 강조할 `nodeIds` 목록을 가진 5-15개 steps를 생성합니다 [understand-anything-plugin/skills/understand-diff/SKILL.md:22](). 언어별 교육 notes를 위해 선택적 `languageLesson`을 포함할 수 있습니다 [understand-anything-plugin/skills/understand-diff/SKILL.md:22](). language directive가 제공되면 tour title과 descriptions가 번역됩니다 [understand-anything-plugin/agents/tour-builder.md:16-20]().

에이전트는 다음 정보를 활용합니다.

*   **Entry Point Candidates**: tour는 일반적으로 가장 중요한 entry points, 특히 `README.md`에서 시작합니다 [understand-anything-plugin/agents/tour-builder.md:74]().
*   **Fan-In Ranking**: fan-in이 높은 nodes는 중요성 때문에 초기에 소개됩니다.
*   **BFS Traversal**: BFS order는 dependencies를 통과하는 자연스러운 진행을 제공합니다.
*   **Tightly Coupled Clusters**: 이러한 groups는 관련 components를 설명하기 위해 하나의 step으로 함께 제시됩니다.
*   **Non-Code File Inventory**: 관련 non-code files는 프로젝트의 완전한 그림을 제공하기 위해 tour에 통합됩니다.
*   **Layers**: architectural layers는 tour steps를 구성하기 위한 상위 수준 구조를 제공합니다.

이 단계의 출력은 `TourStep` 객체 목록이며, 이후 최종 `KnowledgeGraph`에 포함됩니다 [understand-anything-plugin/skills/understand-diff/SKILL.md:22]().
