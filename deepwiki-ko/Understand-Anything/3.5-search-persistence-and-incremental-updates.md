이 페이지는 Knowledge Graph 내부 검색, 그래프와 관련 메타데이터의 디스크 영속화, 재분석 비용을 최소화하기 위한 증분 업데이트 메커니즘을 자세히 설명합니다. 여기에는 `SearchEngine`(fuzzy search), `SemanticSearchEngine`(embedding 기반 검색), 영속성 계층(`readGraph`, `writeGraph`, `sanitiseFilePaths`), fingerprinting(`FileFingerprint`, `ChangeLevel`), `change-classifier`(`SKIP`/`PARTIAL_UPDATE`/`ARCHITECTURE_UPDATE`/`FULL_UPDATE`), 그리고 stale 상태 감지가 포함됩니다.

## 영속성 계층

영속성 계층은 `KnowledgeGraph`와 관련 메타데이터를 디스크에 읽고 쓰는 역할을 담당합니다. 프로젝트의 모든 영속 데이터는 프로젝트 루트의 `.understand-anything/` 디렉터리 안에 저장됩니다. 이 디렉터리는 ESLint 설정에서 기본적으로 무시됩니다 [eslint.config.mjs:13-13]().

### 디렉터리 구조

생성된 모든 파일은 프로젝트 루트 안의 `.understand-anything/` 디렉터리에 저장됩니다.
```
.understand-anything/
├── knowledge-graph.json    // The main KnowledgeGraph
├── domain-graph.json       // The Domain Graph (if generated)
├── meta.json               // Metadata about the last analysis
├── fingerprints.json       // Structural fingerprints of analyzed files
└── config.json             // Project-specific configuration (e.g., auto-update preference)
```

### 파일 경로 정리

영속성의 중요한 측면은 `sanitiseFilePaths` [understand-anything-plugin/packages/core/src/persistence/index.ts:38-67]()입니다. 이 함수는 민감한 사용자별 정보(예: `/Users/alice/company/src/auth.ts`)를 포함할 수 있는 절대 파일 경로가 `knowledge-graph.json`에 쓰이기 전에 상대 경로(예: `src/auth.ts`)로 변환되도록 보장합니다. 경로가 절대 경로이지만 프로젝트 루트 밖에 있으면 정보 유출을 최소화하기 위해 파일 이름만 유지합니다. 이는 기본 `knowledge-graph.json`과 `domain-graph.json` 모두에 적용됩니다 [understand-anything-plugin/packages/core/src/persistence/index.ts:76-76](), [understand-anything-plugin/packages/core/src/persistence/index.ts:154-154]().

## 증분 업데이트와 Staleness 감지

이 시스템은 Knowledge Graph에 대해 증분 업데이트를 수행하도록 설계되어 전체 재분석 필요성을 최소화하고, 그 결과 token 비용과 처리 시간을 줄입니다. 이는 fingerprinting, 변경 분류, 자동화 hook의 조합으로 달성됩니다.

### Staleness 감지

Staleness 감지는 주로 `hooks.json`에 정의된 `SessionStart` hook이 처리합니다 [understand-anything-plugin/hooks/hooks.json:14-25](). 이 hook은 `.understand-anything/meta.json`에 저장된 `gitCommitHash`가 현재 `HEAD` commit과 다른지 확인합니다. 서로 다르고 `config.json`에서 `autoUpdate`가 활성화되어 있으면, LLM에게 `auto-update-prompt.md`를 읽고 실행하도록 지시하여 auto-update workflow를 트리거합니다.

### Auto-Update Workflow (`auto-update-prompt.md`)

`auto-update-prompt.md` 문서는 Knowledge Graph를 증분 업데이트하기 위한 상세한 다단계 프로세스를 설명합니다. 이 prompt는 Git commit hook(`PostToolUse`) 또는 그래프가 stale 상태일 때 session start에 의해 트리거됩니다.

#### Phase 0 — Pre-flight (Zero Token Cost)

이 단계는 LLM token을 전혀 사용하지 않고 초기 검사와 파일 필터링을 수행합니다.
1.  **그래프 존재 여부 확인**: `knowledge-graph.json`과 `meta.json`이 존재하는지 검증합니다.
2.  **Commit Hash 비교**: 저장된 `gitCommitHash`를 현재 `HEAD`와 비교합니다. 일치하면 그래프는 최신 상태입니다.
3.  **변경 파일 감지**: `git diff`를 사용해 마지막 분석 이후 변경된 파일을 식별합니다.
4.  **소스 파일 필터링**: 변경된 파일 중 관련 소스 코드 확장자(예: `.ts`, `.py`, `.go`)만 포함하도록 필터링합니다.
5.  **`.understandignore` 제외**: `.understandignore` 규칙을 적용해 분석하지 않아야 하는 파일을 추가로 걸러내며, test file이나 vendored code 같은 무시된 경로의 불필요한 처리를 방지합니다. 이 단계는 동적으로 생성된 Node.js script(`ignore-filter.mjs`)를 사용해 `@understand-anything/core`의 `createIgnoreFilter` 유틸리티를 활용합니다 [understand-anything-plugin/hooks/auto-update-prompt.md:55-79]().

#### Phase 1 — Structural Fingerprint Check (Zero LLM Tokens)

이 단계는 결정론적 structural fingerprinting을 사용해 변경 범위를 판단하는 데 중요합니다. 포함되는 작업은 다음과 같습니다.
1.  **Fingerprint 로딩**: `fingerprints.json`을 읽습니다 [understand-anything-plugin/hooks/auto-update-prompt.md:101-101]().
2.  **Content Hashing**: 변경된 각 파일에 대해 SHA-256 content hash를 계산합니다. hash가 저장된 값과 일치하면 해당 파일은 `NONE`(변경 없음)으로 간주됩니다.
3.  **구조 요소 추출**: regex pattern을 사용해 구조 요소(functions, classes, imports, exports)를 추출합니다. 이는 전체 Tree-Sitter 분석과 비교했을 때 단순화된 결정론적 접근입니다.
4.  **Change Level 분류**: 추출된 요소를 저장된 fingerprint와 비교해 변경을 `NONE`, `COSMETIC`, `STRUCTURAL`로 분류합니다.
5.  **신규/삭제 파일 처리**: 신규 파일은 `STRUCTURAL`로 분류됩니다. fingerprint에는 있지만 디스크에는 없는 삭제 파일도 `STRUCTURAL`입니다.
6.  **전체 업데이트 결정**: 집계된 파일 변경 수준에 따라 `UpdateDecision`이 내려지며, 가능한 값은 `SKIP`, `PARTIAL_UPDATE`, `ARCHITECTURE_UPDATE`, `FULL_UPDATE`입니다. 결과는 `change-analysis.json`에 기록됩니다 [understand-anything-plugin/hooks/auto-update-prompt.md:121-121]().

### Fingerprinting (`FileFingerprint`, `FingerprintStore`)

`fingerprint.ts` 모듈은 파일 fingerprint를 생성하고 관리하기 위한 데이터 구조와 로직을 정의합니다.

-   `FileFingerprint`: 단일 파일의 structural fingerprint를 나타냅니다 [understand-anything-plugin/packages/core/src/fingerprint.ts:10-20](). 포함 항목은 다음과 같습니다.
    -   `filePath`: 파일의 상대 경로입니다.
    -   `contentHash`: 파일 내용의 SHA-256 hash입니다.
    -   `functions`, `classes`, `imports`, `exports`: 이러한 구조 요소들의 이름/식별자 배열입니다.
    -   `totalLines`: 파일의 총 줄 수입니다.
    -   `hasStructuralAnalysis`: 파일이 구조 요소에 대해 완전히 분석되었는지를 나타내는 boolean입니다.
-   `FingerprintStore`: `FileFingerprint` 객체 모음과 `version`, `gitCommitHash`, `generatedAt` 같은 메타데이터입니다 [understand-anything-plugin/packages/core/src/fingerprint.ts:22-27]().

`buildFingerprints` 함수 [understand-anything-plugin/skills/understand/build-fingerprints.mjs:10-10]()(auto-update prompt에서 사용됨)는 이러한 fingerprint 생성을 담당합니다. 이 함수는 파일 내용을 읽고, hash를 계산하며, regex와 가능한 경우 더 정확한 분석을 위한 Tree-Sitter를 조합해 구조 요소를 추출합니다.

### 변경 분류 (`ChangeLevel`, `classifyUpdate`)

`change-classifier.ts` 모듈에는 `classifyUpdate` 함수가 포함되어 있습니다. 이 함수는 `ChangeAnalysis`(신규, 삭제, 구조적으로 변경된 파일의 세부 정보)와 그래프의 전체 파일 수를 받아 적절한 업데이트 동작을 결정합니다.

-   `ChangeLevel`: 파일 변경을 분류하기 위해 내부적으로 사용되는 enum 유사 개념입니다(`types.ts`에 명시적 enum으로 정의되어 있지는 않음).
    -   `NONE`: 변경 없음.
    -   `COSMETIC`: 내용은 변경되었지만 구조 요소(functions, classes, imports, exports)는 동일합니다.
    -   `STRUCTURAL`: 구조 요소가 변경되었습니다.

-   `UpdateDecision`: `classifyUpdate`의 출력으로, 권장 동작을 지정합니다 [understand-anything-plugin/packages/core/src/change-classifier.ts:4-10]().
    -   `action`: `SKIP`, `PARTIAL_UPDATE`, `ARCHITECTURE_UPDATE`, 또는 `FULL_UPDATE`.
    -   `filesToReanalyze`: 재분석이 필요한 파일 목록입니다.
    -   `rerunArchitecture`: architecture analysis를 다시 실행해야 하는지를 나타내는 boolean입니다.
    -   `rerunTour`: tour를 다시 생성해야 하는지를 나타내는 boolean입니다.
    -   `reason`: 결정에 대한 사람이 읽을 수 있는 설명입니다.

`classifyUpdate` 함수 [understand-anything-plugin/packages/core/src/change-classifier.ts:21-87]()는 decision matrix를 구현합니다.
-   **`SKIP`**: 구조적 변경이 감지되지 않은 경우(모든 파일이 `NONE` 또는 `COSMETIC`).
-   **`PARTIAL_UPDATE`**: 일부 파일에 `STRUCTURAL` 변경이 있지만 디렉터리 구조가 안정적이고 구조적 변경 수가 적은 경우(10개 이하 파일).
-   **`ARCHITECTURE_UPDATE`**: `detectDirectoryChanges` [understand-anything-plugin/packages/core/src/change-classifier.ts:94-114]()를 통해 신규 또는 삭제 디렉터리가 감지되거나, 구조적으로 변경된 파일이 10개를 초과하는 경우입니다. 이는 architecture analysis 재실행을 트리거합니다.
-   **`FULL_UPDATE`**: 많은 구조적 변경이 발생한 경우(30개 초과 파일 또는 전체 그래프 파일의 50% 초과)입니다. 이는 대규모 refactor를 시사하며 완전한 rebuild를 권장합니다.

## 검색 엔진

시스템은 fuzzy search와 semantic search라는 두 가지 주요 검색 메커니즘을 제공합니다.

### Fuzzy Search (`SearchEngine`)

`SearchEngine` 클래스(snippet에 명시적으로 제공되지는 않았지만 dashboard에서의 사용으로 암시됨)는 일반적으로 [Fuse.js](https://fusejs.io/) 같은 라이브러리를 활용해 빠른 client-side fuzzy search를 수행합니다. 이 클래스는 `name`, `summary`, `tags` 같은 `GraphNode` 속성을 색인합니다.

사용자가 search bar에 입력하면 `SearchEngine`은 오타나 부분 일치가 있어도 query와의 관련성을 기준으로 node를 빠르게 필터링하고 순위를 매깁니다. 이는 dashboard의 대화형 필터링에 흔히 사용됩니다.

### Semantic Search (`SemanticSearchEngine`)

`SemanticSearchEngine`(역시 사용 방식으로 암시됨)은 node 설명 또는 summary의 의미적 유사도에 기반해 검색을 수행합니다. 이를 위해 각 `GraphNode`에 대해 사전 계산된 embedding이 필요합니다. query가 만들어지면 query 자체를 embedding한 다음, 저장된 node embedding을 대상으로 cosine similarity search를 수행합니다. 유사도 점수가 높은 node가 결과로 반환됩니다. 이를 통해 사용자는 정확한 이름이나 keyword를 모르더라도 찾고 있는 내용을 자연어로 설명하여 관련 code entity를 찾을 수 있습니다.

이 engine은 embedding을 사용해 "Natural Language Space"(사용자 query)를 "Code Entity Space"(`GraphNode`)와 연결합니다.

## 비코드 파서와 구조 분석

핵심 `TreeSitterPlugin`이 code language를 처리하는 한편, 시스템은 다양한 비코드 파일 형식을 위한 특화 파서도 포함합니다. 이러한 파서들은 `DefinitionInfo`, `ServiceInfo`, `EndpointInfo`, `StepInfo`, `ResourceInfo`를 추출하여 `StructuralAnalysis`에 기여합니다 [understand-anything-plugin/packages/core/src/types.ts:129-160]().

이러한 파서의 예는 다음과 같습니다.
-   **`SQLParser`**: SQL 파일에서 `table`, `view`, `index` 정의를 추출합니다 [understand-anything-plugin/packages/core/src/plugins/parsers/sql-parser.ts:26-73]().
-   **`GraphQLParser`**: GraphQL schema file에서 `type`, `input`, `enum`, `interface`, `union`, `scalar` 정의와 `Query`/`Mutation`/`Subscription` endpoint를 추출합니다 [understand-anything-plugin/packages/core/src/plugins/parsers/graphql-parser.ts:25-53](), [understand-anything-plugin/packages/core/src/plugins/parsers/graphql-parser.ts:58-92]().
-   **`RubyExtractor`**: 주로 code extractor이지만, property를 위한 `attr_accessor` 같은 언어별 구조 요소가 어떻게 처리되는지 보여줍니다 [understand-anything-plugin/packages/core/src/plugins/extractors/ruby-extractor.ts:15-15](), [understand-anything-plugin/packages/core/src/plugins/extractors/ruby-extractor.ts:100-164]().

이 파서들은 `AnalyzerPlugin` 인터페이스 [understand-anything-plugin/packages/core/src/types.ts:195-203]()를 구현하고, 각자의 특정 발견 사항으로 `StructuralAnalysis` 객체를 채웁니다. 이 객체는 이후 `KnowledgeGraph`를 구축하는 데 사용됩니다.
