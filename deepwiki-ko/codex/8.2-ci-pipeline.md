`rust-ci` workflow는 변경된 경로를 감지하고 관련 job만 실행하여 빠른 PR feedback에 최적화되어 있습니다. 주로 linting, formatting, 경량 검사를 처리합니다.

## 변경 감지(`changed` job)

`changed` job은 `git diff --name-only`를 사용해 PR base와 head 사이의 diff를 검사합니다 [`.github/workflows/rust-ci.yml:38-38`](). 이 job은 downstream job을 gate하기 위해 boolean output을 내보냅니다.

| Output | 로직 / 경로 |
|--------|--------------|
| `codex` | `codex-rs/*` [`.github/workflows/rust-ci.yml:49-49`]() |
| `argument_comment_lint` | `codex-rs/*`, `tools/argument-comment-lint/*`, `justfile`, `defs.bzl`, `*.tpl` [`.github/workflows/rust-ci.yml:50-51`]() |
| `argument_comment_lint_package` | `tools/argument-comment-lint/*`, `rust-ci.yml`, `rust-ci-full.yml` [`.github/workflows/rust-ci.yml:52-52`]() |
| `workflows` | `.github/*` [`.github/workflows/rust-ci.yml:53-53`]() |

## Lint/Test 매트릭스

Codex는 다양한 target matrix 전반에서 빌드를 검증합니다. `rust-ci-full.yml` workflow와 `rust-release.yml`은 지원되는 플랫폼 구성을 정의합니다.

| OS | Target Triple | Profile | 사용 사례 |
|----|---------------|---------|----------|
| macOS 15 | `aarch64-apple-darwin` | `dev` / `release` | 기본 Apple Silicon [`.github/workflows/rust-ci-full.yml:178-180`]() |
| macOS 15 | `x86_64-apple-darwin` | `dev` / `release` | Legacy Intel Mac [`.github/workflows/rust-ci-full.yml:181-183`]() |
| Ubuntu 24.04 | `x86_64-unknown-linux-musl` | `dev` / `release` | 정적 Linux 바이너리 [`.github/workflows/rust-ci-full.yml:184-186`]() |
| Ubuntu 24.04 | `aarch64-unknown-linux-musl` | `dev` / `release` | ARM64 Linux [`.github/workflows/rust-ci-full.yml:190-193`]() |
| Windows | `x86_64-pc-windows-msvc` | `release` | 표준 Windows [`.github/workflows/rust-release-windows.yml:27-33`]() |

### 의존성 및 보안 감사
- **Cargo Shear**: `Cargo.toml`에서 사용하지 않는 의존성을 감지하고 제거합니다 [`.github/workflows/rust-ci.yml:104-104`]().
- **Verify Manifests**: Python script는 workspace 설정이 repository 전반에서 올바르게 상속되는지 검증합니다 [`.github/workflows/ci.yml:21-21`]().
- **TUI Boundaries**: `codex-tui`가 `codex-core`를 직접 import하지 않도록 보장합니다 [`.github/workflows/ci.yml:24-24`]().
- **Issue Automation**: `issue-labeler.yml`, `issue-deduplicator.yml` 같은 workflow는 AI를 사용해 repository를 triage하고 정리합니다 [`.github/workflows/issue-labeler.yml:10-19`](), [`.github/workflows/issue-deduplicator.yml:10-19`]().

## Argument Comment Linting

Codex는 `tools/argument-comment-lint/`의 사용자 지정 Dylint 기반 도구를 사용해 literal argument에 대한 특정 스타일을 강제합니다. 이는 `true`, `false`, `None` 같은 positional argument 앞에 parameter name과 일치하는 주석(예: `/*is_active*/ true`)이 오도록 보장합니다.

### 구현 세부 사항
- **Toolchain**: `rustc-dev`, `rust-src`, `llvm-tools-preview`가 포함된 특정 nightly toolchain(`nightly-2025-09-18`)이 필요합니다 [`.github/workflows/rust-ci.yml:123-129`]().
- **Dylint**: `cargo-dylint`와 `dylint-link` version `5.0.0`을 사용합니다 [`.github/workflows/rust-ci.yml:112-113`]().
- **Python Wrappers**: `run.py`와 `wrapper_common.py`가 여러 빌드 환경에서 linter 실행을 조정합니다 [`.github/workflows/rust-ci.yml:148-150`]().
- **Bazel Integration**: Windows가 아닌 플랫폼에서는 `--config=argument-comment-lint` flag와 함께 `run-bazel-ci.sh`를 통해 실행됩니다 [`.github/workflows/rust-ci-full.yml:138-145`]().

출처: [`.github/workflows/rust-ci.yml:106-156`](), [`.github/workflows/rust-ci-full.yml:56-157`]()

## MUSL Cross-Compilation 설정

정적 Linux 바이너리는 distribution 전반에서 최대 이식성을 보장하기 위해 MUSL target을 사용해 생성됩니다. `codex-cli` npm 패키지는 이러한 target을 각 platform package에 명시적으로 매핑합니다 [`codex-cli/bin/codex.js:15-22`]().

### MUSL 빌드 파이프라인
`rust-release.yml` workflow는 MUSL 빌드를 위한 hermetic 환경을 설정합니다.
1. **Runner Selection**: 특정 Linux x64 및 ARM64 runner를 사용합니다 [`.github/workflows/rust-release.yml:104-122`]().
2. **Hermetic Home**: MUSL 빌드를 위한 hermetic Cargo home을 생성합니다 [`.github/workflows/rust-release.yml:174-180`]().
3. **Platform Mapping**: CLI는 Linux 환경에서 `x86_64-unknown-linux-musl` 및 `aarch64-unknown-linux-musl`을 식별합니다 [`codex-cli/bin/codex.js:32-36`]().

## SDK 테스트

`sdk.yml` workflow는 Python 및 TypeScript SDK를 모두 검증합니다.

### SDK 검증 파이프라인
- **Python SDK**: `python:3.12-slim` Docker container 안에서 실행되어 `manylinux` runtime에 대한 의존성 해석을 보장합니다 [`.github/workflows/sdk.yml:28-34`](). 환경 관리에는 `uv`를, 테스트에는 `pytest`를 사용합니다 [`.github/workflows/sdk.yml:37-41`]().
- **TypeScript SDK**: `pnpm`을 사용해 빌드하고 테스트합니다 [`.github/workflows/sdk.yml:145-154`]().
- **Bazel Integration**: TypeScript SDK 테스트는 Bazel을 통해 빌드된 `codex` 바이너리에 의존합니다 [`.github/workflows/sdk.yml:93-100`]().

## 릴리스와 배포

`rust-release` workflow는 `rust-v*.*.*`와 일치하는 tag로 트리거되는 production artifact 생성을 관리합니다 [`.github/workflows/rust-release.yml:15-15`]().

### 릴리스 매트릭스 및 번들
바이너리는 배포를 위해 "bundle"로 그룹화됩니다.
- **Primary**: `codex`, `codex-responses-api-proxy`, 그리고 Linux의 경우 `bwrap`를 포함합니다 [`.github/workflows/rust-release.yml:83-120`]().
- **App-Server**: `codex-app-server` JSON-RPC bridge를 포함합니다 [`.github/workflows/rust-release.yml:89-126`]().
- **Helpers (Windows)**: `codex-windows-sandbox-setup` 및 `codex-command-runner`를 포함합니다 [`.github/workflows/rust-release-windows.yml:43-51`]().

### 서명 절차
- **Windows**: `windows-code-sign` composite action을 통해 Azure Trusted Signing을 사용합니다 [`.github/actions/windows-code-sign/action.yml:32-73`]().
- **Tag Validation**: 릴리스 실행은 `codex-rs/Cargo.toml`의 버전과 일치하는 tag에서 시작되어야 합니다 [`.github/workflows/rust-release.yml:36-48`]().
