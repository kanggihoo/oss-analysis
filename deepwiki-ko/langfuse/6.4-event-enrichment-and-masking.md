이 페이지는 Langfuse ingestion pipeline 내 event processing, enrichment, masking logic을 설명합니다. Event가 API를 통해 ingest되고 S3에서 retrieve된 뒤에는 ClickHouse에 기록되기 전에 transformation, validation, enrichment를 거칩니다. 이 process는 주로 `IngestionService`와 OpenTelemetry data를 위한 specialized processor가 처리합니다.

## Processing Flow 개요

## Ingestion Masking(Enterprise)

Langfuse는 ClickHouse에 persist되기 전 ingestion 단계에서 민감한 data를 masking하는 기능을 지원합니다. 이는 `@langfuse/ee` package에 구현되어 worker에서 사용하는 enterprise feature입니다.

1. **Header Propagation**: API layer는 `LANGFUSE_INGESTION_MASKING_PROPAGATED_HEADERS`에 정의된 특정 header를 extract하고 queue payload를 통해 worker로 propagate합니다.
2. **Execution**: `isIngestionMaskingEnabled`가 true를 반환하면 masking logic이 `applyIngestionMasking`을 통해 event body에 적용됩니다.

## Event Record Enrichment

`IngestionService.createEventRecord` function은 loose `EventInput`을 strict `EventRecordInsertType`으로 변환하는 단일 지점 역할을 합니다 [worker/src/services/IngestionService/index.ts:211-214](). 몇 가지 중요한 enrichment step을 수행합니다.

*   **Prompt Lookup**: `PromptService`를 사용해 name과 version으로 prompt reference를 resolve합니다 [worker/src/services/IngestionService/index.ts:226-233]().
*   **Model & Usage**: `findModel`을 통해 model을 resolve하고 SDK가 제공하지 않은 경우 tokenization 또는 cost를 계산합니다 [worker/src/services/IngestionService/index.ts:235-236]().
*   **Metadata Flattening**: `convertRecordValuesToString`을 통해 nested metadata를 string record로 변환합니다 [worker/src/services/IngestionService/utils.ts:45-54]().
*   **Timestamp Normalization**: JS Date를 ClickHouse-compatible format으로 변환하는 것처럼 모든 incoming timestamp를 normalize합니다 [worker/src/services/IngestionService/index.ts:12-15]().

## Model Matching 및 Pricing

시스템은 cost를 계산하기 위해 event에 제공된 model name을 internal `Model` definition으로 resolve합니다.

### Model Resolution Flow
`findModel`은 model name을 database record와 관련 pricing tier로 resolve하는 기본 entry point입니다. 성능 최적화를 위해 multi-layer cache(Local L1 + Redis L2)를 활용합니다.

1. **Local Cache(L1)**: `modelMatchLocalCache`를 확인합니다.
2. **Redis Cache(L2)**: L1 miss가 발생하면 `getModelWithPricesFromRedis`를 사용해 Redis를 확인합니다.
3. **Postgres Lookup**: 시스템은 `Model` table을 query하여 model name을 regex pattern과 match합니다.
4. **Pricing Tiers**: Model은 다양한 usage type(input, output, total)에 대한 `Price` definition을 포함하는 `PricingTier` record와 연결됩니다.

**Diagram: Model Matching Logic**

```mermaid
graph TD
    [Start] --> [modelMatchLocalCache.getOrLoad]
    [modelMatchLocalCache.getOrLoad] -- "Miss" --> [getModelWithPricesFromRedis]
    [getModelWithPricesFromRedis] -- "Miss" --> [findModelInPostgres]
    [findModelInPostgres] --> [Prisma.Model.findMany]
    [Prisma.Model.findMany] -- "Found" --> [findPricingTiersForModel]
```

### Pricing Tiers 및 Cost Calculation
`IngestionService`는 user-provided cost와 resolved pricing을 기반으로 계산된 cost를 비교해 usage cost를 계산합니다 [worker/src/services/IngestionService/tests/calculateTokenCost.unit.test.ts:143-152](). User-provided cost는 항상 system-calculated cost보다 우선합니다 [worker/src/services/IngestionService/tests/calculateTokenCost.unit.test.ts:180-183]().

## Event Attribute Extraction

Enrichment 중에는 raw event body에서 특정 domain-level attribute가 extract됩니다.

- **Tool Extraction**: `normalizeToolsForObservation`은 observation payload에서 tool definition과 call을 parse합니다 [worker/src/services/IngestionService/index.ts:50-51]().
- **JSON Deep Parsing**: `deepParseJson` 및 `deepParseJsonIterative`는 input/output의 nested stringified JSON을 처리합니다 [packages/shared/src/utils/json.ts:61-79](). 여기에는 Python literal(`True`, `False`, `None`)을 JSON으로 변환하여 Python-style dictionary(예: LangChain에서 온 것)를 처리하는 `tryParsePythonDict`가 포함됩니다 [packages/shared/src/utils/json.ts:9-41]().
- **Usage Schema Transformation**: 시스템은 다양한 usage format(OpenAI completion vs. OpenAI response vs. standard Langfuse)을 처리하고 Zod transform을 사용해 unified internal `Usage` schema로 map합니다 [packages/shared/src/server/ingestion/types.ts:45-69]().
- **Score Inflation**: `validateAndInflateScore`는 `configId` reference를 `ScoreConfig`로 resolve하여 data type(NUMERIC, CATEGORICAL, BOOLEAN, TEXT, CORRECTION)을 enforce하고 categorical label을 numeric value로 map합니다 [worker/src/services/IngestionService/index.ts:43-43]().

## Event Propagation(V4 Architecture)

ingestion system은 observations가 initially staging table에 기록된 뒤 나중에 primary events table로 propagate되는 "events-first" architecture를 점점 더 많이 사용하고 있습니다.

*   **Staging Table**: `observations_batch_staging`은 12시간 TTL이 있는 temporary buffer 역할을 합니다 [packages/shared/clickhouse/scripts/dev-tables.sh:81-130](). Efficient batch processing을 가능하게 하기 위해 3분 partition을 사용합니다 [packages/shared/clickhouse/scripts/dev-tables.sh:121]().
*   **Propagation Job**: `handleEventPropagationJob`은 이러한 partition을 sequentially 처리하는 BullMQ worker입니다 [worker/src/features/eventPropagation/handleEventPropagationJob.ts:58-60](). Staging observations를 trace metadata와 join하여 full `events_full` record를 구성합니다 [worker/src/features/eventPropagation/handleEventPropagationJob.ts:140-185]().
*   **Cursor Management**: Job은 data가 누락되거나 double-process되지 않도록 `LAST_PROCESSED_PARTITION_KEY`를 사용해 Redis에서 progress를 추적합니다 [worker/src/features/eventPropagation/handleEventPropagationJob.ts:15-50]().
